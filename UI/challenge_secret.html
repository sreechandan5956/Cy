üß© Key Concepts

‚û§ HTML Source Code Inspection

Developers sometimes leave sensitive information in HTML comments or hidden fields. The page source shows the raw HTML sent by the server, which may contain information not visible in the rendered page.

  • HTML comments use <!-- comment --> syntax
  • Comments are ignored by browsers but visible in source
  • Hidden input fields have type="hidden"
  • JavaScript files may contain hardcoded values

How to view: Right-click ‚Üí "View Page Source" or press Ctrl+U (Cmd+Option+U on Mac)

‚û§ Base64 Encoding

Base64 is a way to encode binary data into ASCII text. It's NOT encryption - it's just encoding! Anyone can decode it without a key.

Characteristics:

  • Uses only A-Z, a-z, 0-9, +, / characters
  • Often ends with = or == for padding
  • Length is always a multiple of 4

Common tools: base64decode.org, CyberChef, or command line tools

‚û§ JWT (JSON Web Tokens)

JWT is a compact way to securely transmit information between parties. It's commonly used for authentication in modern web applications.

JWT Structure:

header.payload.signature
(Base64).(Base64).(Signature)

Three Parts:

  • Header: Specifies the algorithm (e.g., HS256) and token type
  • Payload: Contains claims - user data, roles, permissions, expiration
  • Signature: Created using header + payload + secret key. Prevents tampering

Important: The header and payload are only Base64 encoded, not encrypted. Anyone can decode and read them. The signature is what makes it secure.

‚û§ JWT Security Concepts

JWT security relies on the secret key used for signing:

  • Without the secret, you can't create valid signatures
  • If the secret is weak, it can be cracked through brute force
  • Once you have the secret, you can forge any token
  • The signature algorithm (HS256, RS256, etc.) matters
‚û§ Browser Developer Tools

  • Elements/Inspector: View and modify HTML/CSS
  • Console: Execute JavaScript, view errors
  • Network: Monitor HTTP requests and responses
  • Application/Storage: View cookies, localStorage, sessionStorage

Access: Press F12 or right-click ‚Üí "Inspect"

‚û§ Privilege Escalation

Privilege escalation is gaining higher access than originally granted. This can happen when:

  • User-controlled data determines permissions (like JWT payload)
  • Access controls are only enforced client-side
  • Tokens can be forged or manipulated
  • Role/permission checks are insufficient

Gaining higher access than originally granted. In this challenge, you'll escalate from "user" to "admin" by modifying the JWT token.

üìù Security Testing Methodology

‚û§ Reconnaissance Phase

The first step in any security assessment is gathering information about the target. This includes:

  • Examining visible functionality and user interfaces
  • Reviewing client-side code (HTML, CSS, JavaScript)
  • Identifying technologies and frameworks in use
  • Mapping out application structure and endpoints

Information can be found in unexpected places - comments, metadata, error messages, or configuration files.

‚û§ Understanding Data Encoding

Web applications often encode data for transmission or storage. Common encoding schemes include:

  • Base64: Converts binary data to ASCII text
  • URL Encoding: Makes data safe for URLs
  • Hex Encoding: Represents bytes as hexadecimal

Key Point: Encoding is not encryption. Encoded data can be decoded by anyone without a key. It's meant for data representation, not security.

‚û§ Token-Based Authentication

Modern applications commonly use tokens instead of traditional sessions:

  • Tokens are issued after successful authentication
  • Clients include tokens in subsequent requests
  • Servers validate tokens to identify users
  • Tokens may contain user data, roles, and permissions

Understanding token structure and validation is essential for security testing.

‚û§ Cryptographic Security

Authentication systems rely on cryptographic principles:

  • Hashing: One-way functions for passwords
  • Signing: Verifying data hasn't been tampered with
  • Encryption: Protecting data confidentiality

The strength of these systems depends on key management. Weak or exposed keys compromise the entire system.

‚û§ Access Control Testing

Applications should enforce proper authorization:

  • Users should only access resources they're permitted to
  • Privilege levels should be validated server-side
  • Client-side controls alone are insufficient
  • Horizontal and vertical privilege escalation should be prevented

Testing involves attempting to access resources or perform actions beyond your assigned privileges.

üîç Common Authentication Vulnerabilities

‚û§ Information Disclosure

Sensitive information exposed through:

  • HTML comments containing credentials or hints
  • Verbose error messages revealing system details
  • Exposed configuration files or backup files
  • Client-side code containing API keys or secrets
‚û§ Weak Cryptographic Implementations

Common issues include:

  • Using weak or default secrets for signing
  • Insufficient key length or complexity
  • Hardcoded secrets in source code
  • Lack of key rotation policies
  • Using deprecated or weak algorithms
‚û§ Insufficient Access Controls

Authorization flaws that allow:

  • Horizontal privilege escalation (accessing other users' data)
  • Vertical privilege escalation (gaining admin access)
  • Insecure Direct Object References (IDOR)
  • Client-side only authorization checks
‚û§ Token Vulnerabilities

Issues specific to token-based systems:

  • Tokens without expiration times
  • Sensitive data stored in token payload
  • Lack of token revocation mechanisms
  • Algorithm confusion attacks
  • Token replay attacks

üõ°Ô∏è Defense Mechanisms

‚û§ For Developers
  • Use strong, random secrets (minimum 256 bits)
  • Store secrets in environment variables, not code
  • Implement token expiration (short-lived tokens)
  • Use refresh tokens for long sessions
  • Validate token claims on the server side
  • Never store sensitive data in JWT payload
  • Implement rate limiting on authentication endpoints
  • Use HTTPS to prevent token interception

üí° Best Practices for Security Testing

‚û§ Systematic Approach
  • Start with passive reconnaissance before active testing
  • Document all findings, even seemingly insignificant ones
  • Test one thing at a time to understand cause and effect
  • Keep detailed notes of your methodology
‚û§ Understanding Before Exploiting
  • Learn how the technology works normally before testing vulnerabilities
  • Read documentation and specifications
  • Understand the security model being used
  • Consider both client-side and server-side perspectives
‚û§ Tool Usage
  • Understand what tools do, don't just run them blindly
  • Manual testing often reveals issues automated tools miss
  • Verify tool results manually when possible
  • Learn to use built-in browser tools before external tools
‚û§ Ethical Considerations
  • Only test systems you own or have explicit permission to test
  • Respect scope limitations and rules of engagement
  • Report vulnerabilities responsibly
  • Consider the impact of your testing on production systems
